# AI 测试执行页布局设计说明

## 目标

在不改变组件结构、状态管理、交互行为和 API 的前提下，仅通过 CSS 调整 AI 测试执行页的布局，使四张核心卡片在桌面端形成稳定、对齐且可随视口伸缩的工作区，并在窄屏下自然切换为单列。

本次覆盖以下四张卡片：

- 执行入口
- 任务队列
- 执行预览
- 执行过程（常规执行时标题为“执行步骤预览”，智能探索时标题为“智能探索执行过程”）

## 现状与根因

当前页面主网格已经采用 `400px minmax(0, 1fr)` 双栏，但执行入口被同时设置为 `height`、`min-height`、`max-height: 314px`。固定高度使内容、字体或可用宽度变化时无法自然扩展，也让左栏上下卡片与右栏卡片的视觉节奏依赖偶然的内容高度。

四张卡片共用基础 Card 结构，但执行页只局部覆盖了标题区下边距，没有明确统一标题区高度、卡片内边距和栏内间距，因此副标题换行或内容差异会造成正文起点不齐。任务列表仅使用固定 `max-height: 700px`，不能跟随视口高度变化：较矮视口容易把页面整体拉长，较高视口则不能充分利用空间。

现有 `1180px` 媒体查询只调整部分子网格，未完整覆盖执行页主网格、执行过程布局和执行预览布局，所以窄屏仍可能保留多栏并产生压缩或横向拥挤。

## 布局规格

### 页面与主网格

- 桌面端（视口宽度大于 `1180px`）保留 `.execution-grid` 的 `400px minmax(0, 1fr)` 两栏结构。
- 左栏固定为 `400px`，右栏占据剩余可用宽度；`minmax(0, 1fr)` 必须保留，避免右栏长内容撑破网格。
- 主网格列间距、`.execution-left` 和 `.execution-right` 的卡片间距统一为 `18px`。
- 左右栏均保持顶部对齐；左栏卡片按内容和可用空间排列，不通过固定高度制造对齐。

### 四卡统一规则

- 四张卡片沿用现有 Card 视觉样式，不新增包装组件或 DOM。
- 卡片内容区使用一致的内边距；以现有 Card 基础内边距为唯一基准，执行页不得为单张核心卡片设置不同内边距。
- 四张卡片的 `.card__header` 使用一致的最小高度和 `14px` 下边距，使标题、副标题区域的正文起点一致；标题或副标题换行时允许标题区自然增高，不裁切文本。
- 标题字号继续使用执行页现有的 `18px`，副标题继续使用现有的 `12px`，不改变文案和层级。
- `.execution-entry-card` 删除 `height: 314px`、`min-height: 314px` 和 `max-height: 314px`，改为内容驱动的自适应高度。
- `.execution-live-card` 可保留现有最小高度以承载过程信息，但不得设置固定高度或阻止内容自然增长。

### 执行预览与执行过程

- 执行预览的状态、任务编号、报告入口、三项摘要和控制台保持现有结构与功能。
- 执行过程的阶段条、步骤选择、截图/画布、详情侧栏和当前滚动能力保持现有结构与功能。
- 桌面端 `.execution-live-layout` 保留 `292px minmax(0, 1fr)`，`.execution-live-preview` 保留 `minmax(0, 1.2fr) 320px`。
- 本次不改变任何条件渲染、点击处理、轮询、选中状态或数据格式化逻辑。

## 响应式规则

断点统一为 `@media (max-width: 1180px)`，在该断点及以下同时执行以下规则：

- `.execution-grid` 切换为 `grid-template-columns: minmax(0, 1fr)`，主页面改为单列。
- `.execution-live-layout` 切换为 `grid-template-columns: minmax(0, 1fr)`，步骤列表位于详情内容之前。
- `.execution-live-preview` 切换为 `grid-template-columns: minmax(0, 1fr)`，主预览位于信息侧栏之前。
- `.execution-preview-head` 保持现有单列规则。
- 左栏不再固定 `400px`，所有卡片宽度跟随容器，且不得产生页面级横向滚动。
- 阶段条保留自身横向滚动，避免压缩阶段节点或改变阶段数据结构。

## 滚动策略

- 页面保持正常文档流，不锁定整页高度，也不为左右栏创建独立的整栏滚动区。
- 任务队列仅让 `.execution-list` 内部滚动，卡片标题、标签页和统计信息保持可见。
- 列表最大高度基于视口计算，采用 `max-height: calc(100vh - <页面固定占用高度>)`；固定占用高度应包含顶部导航、流程条、页面间距、任务队列标题区和标签区，并设置合理的最小高度，确保矮屏仍能看到至少一条完整任务。
- 视口较高时列表可增长以利用空间，视口较矮时列表收缩并出现纵向滚动；不得继续使用与视口无关的固定 `700px` 作为唯一上限。
- 执行步骤列表现有 `max-height: 520px; overflow: auto`、阶段条横向滚动、控制台及预览区域现有滚动行为保持不变。
- 所有内部滚动容器使用 `overscroll-behavior: contain`，避免滚动到边界时意外带动整页；不隐藏原生滚动条。

## 无障碍

- 不改变现有标签页语义、按钮元素、ARIA 标签、焦点顺序和键盘操作路径。
- 内部滚动不得裁切键盘焦点轮廓；滚动容器内获得焦点的任务或步骤必须能被浏览器滚动到可视区域。
- 响应式重排只通过 CSS 网格完成，视觉顺序与 DOM 顺序一致：执行入口、任务队列、执行预览、执行过程。
- `prefers-reduced-motion` 的现有处理保持有效，本次不新增动画。
- 文本允许换行，不使用固定高度、截断或隐藏溢出来维持表面对齐。

## 风险与边界

- 视口高度计算中的固定占用值若过大，会让任务列表过短；若过小，会导致页面整体产生额外纵向滚动。实现时必须在常见桌面高度和矮屏高度下实测。
- 统一标题最小高度可能因副标题换行而自然增高，这是防止裁切的预期行为；不得用固定高度或 `overflow: hidden` 强制对齐。
- `1180px` 断点会改变信息密度，但不改变元素顺序、数据和交互。
- 本次范围严格限制为执行页 CSS。不得修改 `ExecutionCenter.tsx`、Card 组件、状态、接口、文案、轮询和业务逻辑，也不得影响其他页面的 Card 样式。

## 验收标准

1. 宽度大于 `1180px` 时，页面稳定呈现 `400px + 1fr` 双栏，左右栏顶部对齐且间距均为 `18px`。
2. 四张核心卡片标题区正文起点视觉一致，卡片内边距一致，标题或副标题换行时无裁切和重叠。
3. 执行入口不再固定为 `314px`，内容增减时高度自然变化。
4. 任务队列随视口高度伸缩，列表内部可滚动，卡片标题和标签页不会随任务滚走。
5. 宽度不大于 `1180px` 时，主网格、执行过程布局和执行预览布局均为单列，页面无非预期横向滚动。
6. 执行预览和执行过程的展示、选择、删除、报告入口、轮询、阶段条、步骤详情和截图预览功能与改动前一致。
7. 键盘焦点可见，滚动容器可正常操作，文本在桌面与窄屏下均不被裁切或覆盖。
8. 改动仅发生在执行页相关 CSS 选择器中，不修改组件、状态、API 或其他页面行为。

## 测试

### 静态检查

- 检查变更仅涉及执行页 CSS。
- 确认 `.execution-entry-card` 不再包含 `314px` 的高度、最小高度或最大高度声明。
- 确认 `1180px` 媒体查询同时覆盖 `.execution-grid`、`.execution-live-layout` 和 `.execution-live-preview`。
- 运行前端构建，确认 CSS 语法和现有 TypeScript 构建均通过。

### 视觉与交互检查

- 在 `1440x900`、`1280x720`、`1180x800`、`1024x768` 和 `390x844` 视口检查布局。
- 在桌面视口确认左栏为 `400px`、右栏自适应、四卡标题区和间距一致。
- 在 `1180px` 及更窄视口确认三组目标网格均切为单列，内容无重叠、裁切和页面级横向滚动。
- 使用足以超过列表可视高度的任务数据，确认只有任务列表内部滚动，标题和标签页保持可见。
- 分别选择常规执行与智能探索任务，确认执行预览、阶段条、步骤选择、详情、截图/画布和报告入口均可用。
- 仅使用键盘遍历标签页、任务、删除按钮、报告按钮和步骤按钮，确认焦点顺序、焦点可见性及滚动跟随正常。
- 开启减少动态效果设置，确认现有运行态动画停止且布局不发生变化。
