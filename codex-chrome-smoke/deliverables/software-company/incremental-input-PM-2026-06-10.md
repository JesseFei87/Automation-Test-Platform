## 1. 产品目标

`CaseToolbox.tsx` 顶部"输入信息"卡片目前承载 5 个字段，其中**所属项目**（自由文本、拼写不一）和**上下文信息**（单一大文本框、LLM 接收的 prompt 无结构）是最影响统计准确性与 LLM 用例生成相关性的两个瓶颈。

本次增量要解决的是：**让"所属项目"从自由文本变成可下拉、可新建、可追溯的标准化项目档案**，并**让"上下文信息"拆成 4 个语义明确的子字段**，从而使 LLM 生成的用例与项目环境/测试上下文的相关性更稳定，并支持按项目维度做用例统计。

业务结果：项目维度用例统计可执行；LLM 生成用例的相关性波动收窄；输入环节的拼写错误、上下文缺漏得到前置约束。

## 2. 用户故事

### P0 故事

1. 作为**测试人员**，我想要在"所属项目"字段里**下拉选择已有项目或新建一个项目**，以便避免手敲导致的拼写不一致，并让生成的用例自动带上该项目的基础信息。
2. 作为**测试负责人**，我想要所有用例的"所属项目"**统一收敛到一张项目档案表**，以便后续能按项目维度做用例数量、覆盖率、缺陷率的统计与对比。
3. 作为**测试人员**，我想要项目里能维护 `base_url` 和 `description`，以便 LLM 在生成用例时知道项目环境，输出更贴合该项目的接口/页面假设。

### P1 故事

4. 作为**测试人员**，我想要把"上下文信息"按 `环境URL / 测试账号 / 排除范围 / 参考文档` 4 个子字段分别填写，以便我每次写用例时不会被一个无结构的大文本框"凭感觉写"。
5. 作为**测试人员**，我想要 4 个上下文子字段**空着就不要污染 prompt**，以便在没有上下文的场景下，LLM 不会收到一堆空标题的噪声段落。

## 3. 需求池

> 优先级与范围口径完全沿用基线第 2 章节，本节仅做产品视角的拆解描述。

### P0 · 所属项目下拉化（5 条）

| ID | 需求 | 价值 |
|---|---|---|
| P0-R1 | `所属项目` 输入控件由 TextField 升级为 MUI Autocomplete | 消除拼写不一致 |
| P0-R2 | 数据源为新增表 `project_profiles`（id / name UNIQUE / base_url / description / 时间戳） | 项目维度可统计 |
| P0-R3 | 提供"+ 新建项目"入口，弹小 Dialog 录入 `name + base_url`，创建后立即出现在下拉中 | 现场可扩展项目档案 |
| P0-R4 | 选中项目后，写入用例草稿/需求池的是 `projectId`（外键），不是字符串 | 数据可关联 |
| P0-R5 | 生成用例的 prompt 注入 `{project_base_url}` `{project_description}` 上下文 | LLM 输出贴合项目环境 |
| P0-R6（附加） | 启动时若 `project_profiles` 为空，自动种子 2 条示例（ICM / DxONE） | 首次开箱即用 |

### P1 · 上下文信息结构化（4 条）

| ID | 需求 | 价值 |
|---|---|---|
| P1-R1 | 拆分为 4 个子字段：`环境URL` / `测试账号` / `排除范围` / `参考文档` | 语义明确 |
| P1-R2 | 4 字段均允许为空；空字段不污染 prompt | 不给 LLM 喂噪声 |
| P1-R3 | 4 字段值随用例保存到 `cases.spec_yaml` 顶层 `context_info`（JSON 文本） | 随用例可追溯 |
| P1-R4 | 4 字段位置在原"上下文信息"位置以 Stack 排列；保持可折叠/展开（默认展开） | 不破坏既有视觉流 |

## 4. UI 状态说明

### 4.1 "所属项目" Autocomplete 状态机

| 状态 | 触发条件 | UI 表现 | 备注 |
|---|---|---|---|
| **idle** | 字段初始进入页面，未聚焦 | 显示 placeholder「请选择或搜索项目」 | — |
| **loading** | Autocomplete 打开且下拉数据尚未加载完成 | 下拉区域显示 loading 指示 | 数据源为空时不要进 loading |
| **typing** | 用户在输入框键入字符 | 输入框显示当前键入文字；下拉按子串匹配过滤 | 支持中英文模糊匹配 |
| **open menu** | 输入框聚焦 / 触发打开 | 下拉浮层显示已有项目列表 | 若已选中项，高亮当前项 |
| **selected** | 用户点击下拉项 | 输入框回填项目 `name`；同步写入 `projectId` | 此时 prompt 注入该项目上下文 |
| **creating new** | 用户点击"+ 新建项目" | 弹出小 Dialog，提交 `name + base_url`，成功后关闭并把新项加入下拉 | `name` 重复要 inline 校验（不要等后端 400） |

> 视觉位置：5 字段顺序保持 `项目 / 关联需求 / 上传 / 需求描述 / 上下文` 不变，不重新排版（基线第 5 节风险 5）。

### 4.2 上下文 4 子字段展开/折叠策略

- **默认状态：展开**（基线第 2 章节 P1 明确"默认展开"）
- 折叠入口：在"上下文信息"标题栏右侧放一个折叠/展开切换控件
- 折叠记忆：本期**不做**"用户上次折叠状态"的持久化记忆（无后端/无 localStorage 约定），每次进入页面默认展开
- 字段排列：Stack 纵向排列 4 个子字段，标签依次为「环境URL」「测试账号」「排除范围」「参考文档」

## 6. 待确认问题

1. **种子项目 ICM / DxONE 的口径**：基线写"插 2 条示例：ICM / DxONE"，但 `base_url` 与 `description` 是否需要真实可访问的占位值（避免 LLM 拿到 `example.com` 失真），还是允许空字符串？
2. **"+ 新建项目" Dialog 的 `description` 字段**：基线提到 Dialog 录入 `name + base_url`，但 `project_profiles.description` 也允许为空；Dialog 是否在 P0 期就需要暴露 `description` 给用户后期再补？
3. **折叠状态是否要本地记忆**：本期默认每次展开，UX 角度看测试人员高频访问"上下文"，但若用户连续折叠两次以上，是否应在下期考虑 localStorage 记忆？
